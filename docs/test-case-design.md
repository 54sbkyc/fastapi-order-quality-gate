# Test Case Design

这份文档说明接口自动化用例是怎么设计出来的。面试时可以用它回答：不是想到哪测到哪，而是围绕业务风险、输入边界、权限控制和状态流转来设计。

## 测试设计目标

- 验证接口返回是否符合预期。
- 验证 OpenAPI 契约是否稳定。
- 验证业务错误码是否稳定。
- 验证数据库副作用是否正确。
- 验证登录态、权限和 token 异常是否被拦截。
- 验证订单状态机不会出现非法流转。

## 等价类设计

| 模块 | 有效等价类 | 无效等价类 | 代表用例 |
| --- | --- | --- | --- |
| 认证 | 新用户名、正确密码 | 重复用户名、错误密码、不存在用户 | 注册成功、重复注册、登录失败 |
| Token | 有效 JWT | 缺失 token、伪造 token、过期 token、不存在用户 token | 访问订单列表 |
| 商品 | 存在商品 ID | 不存在商品 ID、负数 ID | 查询商品详情 |
| 订单 | 已登录、商品存在、库存充足、数量合法 | 未登录、商品不存在、库存不足、数量越界 | 创建订单 |
| 库存一致性 | 最后一件库存只被一个用户买走 | 两个陈旧会话都以为库存充足 | 防止库存超卖 |
| 接口契约 | 核心路径、中文文档、响应码、错误结构稳定 | 路径漏文档、错误响应未声明 | OpenAPI 契约测试 |
| 订单权限 | 订单属于当前用户 | 访问其他用户订单 | 查询订单详情 |
| 状态流转 | `created -> paid`、`created -> cancelled` | 重复支付、取消已支付订单、支付已取消订单 | 订单状态接口 |

## 边界值设计

订单数量使用 `Field(gt=0, le=100)` 约束，因此重点边界是：

| 输入 | 类型 | 预期 |
| ---: | --- | --- |
| `0` | 下边界外 | `VALIDATION_ERROR` |
| `1` | 下边界内 | 可创建订单 |
| `100` | 上边界内 | 请求结构合法，是否成功取决于库存 |
| `101` | 上边界外 | `VALIDATION_ERROR` |
| `-1` | 明显非法 | `VALIDATION_ERROR` |

## 权限和 Token 设计

受保护接口统一通过 `Authorization: Bearer <token>` 访问。测试覆盖这些风险：

- 未带 token：应返回 `NOT_AUTHENTICATED`。
- token 格式错误：应返回 `INVALID_TOKEN`。
- token 签名被伪造：应返回 `INVALID_TOKEN`。
- token 已过期：应返回 `INVALID_TOKEN`。
- token 合法但用户不存在：应返回 `INVALID_TOKEN`。
- 用户访问他人订单：应返回 `FORBIDDEN_ORDER_ACCESS`。

这些用例能证明项目不是只测正常登录，而是覆盖了真实接口常见的鉴权失败场景。

## 状态流转设计

订单状态机：

| 当前状态 | 操作 | 目标状态 | 是否允许 |
| --- | --- | --- | --- |
| `created` | pay | `paid` | 是 |
| `created` | cancel | `cancelled` | 是 |
| `paid` | cancel | - | 否 |
| `cancelled` | pay | - | 否 |
| `paid` | pay | - | 否 |

测试重点：

- 合法状态流转返回更新后的订单状态。
- 非法状态流转返回 `INVALID_ORDER_STATE`。
- 取消订单后要恢复商品库存。

## 数据库断言设计

只断言响应不够，因为接口返回成功不代表数据真的正确变化。这个项目会在关键路径检查数据库：

- 创建订单后，订单记录应落库。
- 创建订单后，商品库存应减少。
- 取消订单后，商品库存应恢复。
- 查询订单时，只返回当前用户自己的订单。
- 两个会话同时读到最后 1 件库存时，最终只能创建 1 个订单，库存不能被超卖。

这类断言能发现“接口响应成功但持久化逻辑错误”的问题。

## 库存一致性设计

库存是订单系统里最容易被问到的业务风险。普通库存不足测试只能证明“当前请求看到库存不足时会拒绝”，但不能证明“两个请求都读到旧库存时不会超卖”。

本项目用确定性的 stale-session 测试覆盖这个风险：

1. 准备一个库存为 `1` 的商品。
2. 打开两个独立 SQLAlchemy Session。
3. 两个 Session 都先读取同一个商品，让它们都缓存 `stock=1`。
4. 第一个 Session 创建订单成功。
5. 第二个 Session 再尝试创建订单，必须返回 `INSUFFICIENT_STOCK`。
6. 最终断言数据库里只有 1 个订单，商品库存为 `0`。

实现侧使用条件更新 `stock >= quantity` 做原子扣减，避免只依赖 Python 内存里的旧库存对象。这个测试没有使用多线程，因此稳定、可重复，适合放进 CI。

## Allure 报告设计

每个 API 测试都要求有 `allure.title`，并按 feature/story 归类。报告里可以看到：

- 用例属于哪个业务模块。
- 每个测试步骤做了什么。
- 失败时属于认证、校验、业务规则还是未知错误。
- 当前测试环境、后端框架、数据库和覆盖率门槛。

`scripts/quality-gate.ps1` 会生成：

- `allure-results/environment.properties`
- `allure-results/categories.json`

## OpenAPI 契约设计

行为测试能证明接口当前能跑通，但接口契约测试能保护“文档和调用约定”不被无意破坏。本项目会读取 `/openapi.json` 并检查：

- 核心路径和 HTTP 方法是否仍存在。
- summary 和 description 是否保持中文，方便面试演示和 Swagger 阅读。
- 订单、认证等核心接口是否声明关键响应码，例如 `401`、`403`、`404`、`409`、`422`。
- 错误响应是否统一为 `{code, message}`。

这类测试适合解释“接口自动化不只是调接口，也能保护接口文档和调用契约”。

## DeepSeek 辅助测试设计

`scripts/generate-test-ideas.py` 会读取本地 OpenAPI 契约，把路径、说明和响应码整理成 prompt。未配置 `DEEPSEEK_API_KEY` 时，它只输出 prompt；配置 key 后才会调用 DeepSeek API。

使用原则：

- DeepSeek 用来辅助发现测试点，不替代测试判断。
- AI 生成内容不直接进质量门禁。
- 有价值的建议需要人工筛选，再写成 Pytest 自动化用例。

## 覆盖率不是唯一目标

覆盖率说明代码有没有被执行，但不能说明断言有没有价值。本项目用覆盖率做最低门槛，同时用业务断言保证测试真正覆盖风险：

- 状态码断言：接口协议是否正确。
- 错误码断言：业务失败是否可识别。
- 响应结构断言：返回格式是否稳定。
- 数据库断言：真实副作用是否正确。
