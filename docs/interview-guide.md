# Interview Guide

这份文档用于面试前快速复习。目标不是背稿，而是让你能把项目讲得像真的做过、想过、验证过。

## 30 秒项目介绍

我做的是一个订单系统双层自动化测试项目。第一层用 FastAPI TestClient 和隔离数据库快速验证业务规则、数据库副作用与并发一致性；第二层用 httpx 通过可配置的 `API_BASE_URL` 请求真实运行环境，验证注册登录、订单生命周期、非法 Token 和用户隔离。两层都接入 GitHub Actions，并保留 Allure、覆盖率和服务日志等失败证据。

## 2 分钟讲解顺序

完整演示步骤可以直接看 [Demo Script](demo-script.md)。

1. 先说业务：用户登录后可以查询商品、创建订单、支付或取消订单。
2. 再说测试：接口测试覆盖状态码、响应结构、业务错误码和数据库副作用。
3. 展示 fixture：测试用户、token、商品数据和隔离数据库都由 fixture 管理。
4. 展示门禁：CI 会自动运行 lint、接口测试、覆盖率、meta 测试、前端构建和浏览器主流程，本地可以加 `-UiSmoke` 复现，并用 `scripts/report-summary.py` 汇总报告证据。
5. 最后说价值：这个项目证明我不只是会点点接口工具，而是理解测试怎么保护真实业务逻辑、接口契约和交付流程。

测试设计材料可以看 [Test Case Design](test-case-design.md)，缺陷分析材料可以看 [Bug Analysis](bug-analysis.md)。

## 面试官可能会问

### 你为什么自己写后端，而不是只测现成接口？

因为自动化测试不应该只会黑盒调用接口。自己写后端后，我能理解接口背后的业务规则，比如订单创建要扣库存，取消订单要恢复库存，已支付订单不能取消。这样设计测试时可以验证真实业务结果，而不是只看 HTTP 状态码。

### 你这个项目的测试重点是什么？

重点是接口自动化的工程化能力。包括 fixture 管理测试数据、参数化覆盖异常和边界场景、Pydantic 校验响应结构、SQLAlchemy 查询数据库副作用，以及用 CI 做覆盖率和回归门禁。

### 为什么已经有 TestClient，还要做真实 HTTP 测试？

`TestClient` 速度快，适合验证服务内部业务和数据库副作用，但它绕过真实端口、网络代理和部署启动过程。真实 HTTP 层只依赖 `API_BASE_URL`，可以指向本地、测试或预发布环境，用少量高价值用例确认服务真的可访问、鉴权链路和核心业务可以跨网络完成。日常默认不跑 E2E，CI 和发布前强制执行。

### 真实环境测试失败后怎么定位？

统一 API Client 会记录每次请求的 URL、状态码、耗时和响应体。失败时 fixture 自动把记录附加到 Allure，并对密码、Authorization 和 Token 脱敏。连接失败还会明确打印目标 `API_BASE_URL`，CI 同时上传后端启动日志。

### 你为什么要做 OpenAPI 契约测试？

行为测试能证明接口当前能跑通，但 OpenAPI 契约测试能保护接口文档和调用约定。比如核心路径不能消失，Swagger 的 summary/description 要保持中文，订单接口要声明 `401`、`403`、`404`、`409`、`422` 等业务响应，错误响应结构要统一为 `{code, message}`。这样前后端联调和接口文档不会被无意改坏。

### 认证失败场景你测了哪些？

我覆盖了未登录访问、错误密码、重复注册、伪造 token、过期 token、token 对应用户不存在等场景。这样能证明测试不是只覆盖登录成功，还覆盖了接口安全里最常见的负向路径。

### Allure 报告你会怎么看？

我会先看 feature/story 分类，确认失败集中在哪个模块；再看每个用例的 step，定位是数据准备、接口调用还是断言失败；最后看 environment 和 categories，确认测试环境、覆盖率门槛以及失败类型，比如认证失败、参数校验失败或业务规则失败。CI 里会上传 `allure-results`，所以即使流水线失败，也能下载 CI 产物复盘。

### CI 失败后你怎么看证据？

我会先看失败步骤判断是 lint、API 测试、meta 测试、前端构建还是 UI 冒烟；然后下载 CI 产物。`allure-results` 看接口失败，`coverage-xml` 看覆盖率，`playwright-report` 看浏览器步骤；UI 启动失败时再看服务日志。本地可以跑 `scripts/report-summary.py` 快速汇总证据。

### 为什么要做数据库断言？

接口返回成功不代表业务真的正确落库。例如创建订单返回 201，但库存可能没有扣减。数据库断言能验证接口对持久化层产生了正确影响，能发现只靠响应断言看不到的问题。

### fixture 在这里解决了什么问题？

fixture 负责准备可复用的测试上下文，比如测试客户端、隔离数据库、注册用户、登录 token、商品种子数据和已创建订单。这样用例之间不依赖执行顺序，也减少了重复登录、重复造数据的代码。

### 你怎么设计订单状态流转测试？

我把订单状态看成一个小型状态机：`created` 可以变成 `paid` 或 `cancelled`，但 `paid` 不能再取消，`cancelled` 不能再支付，`paid` 也不能重复支付。测试里既覆盖合法路径，也用参数化覆盖非法转换。

### 你怎么考虑库存超卖问题？

我补了一个确定性的 stale-session 测试：库存只有 1 时，两个独立数据库 Session 都先读到 `stock=1`，第一个会话下单成功后，第二个会话再下单必须返回 `INSUFFICIENT_STOCK`。这个测试不用多线程，所以稳定适合进 CI；实现上用数据库条件更新 `stock >= quantity` 做原子扣减，避免只依赖内存里的旧库存对象。

### 两个请求同时取消订单会怎样？

我也用两个陈旧 Session 模拟了并发取消。支付和取消使用带 `status=created` 条件的数据库更新，只有一个请求能成功。取消后的库存恢复和状态提交在同一事务，所以第二个请求会得到 `INVALID_ORDER_STATE`，库存不会重复增加。

### 前端里的测试数和覆盖率可靠吗？

不是手工写死的。质量门禁会先清理旧 Allure 结果，再从本次 Allure JSON 和 `coverage.xml` 生成 `quality-summary.json`。React 看板读取这个快照，Playwright 还会比较页面值和 JSON 是否一致。这个设计修复过本地结果累积导致 46 条用例被误报成 672 个结果的问题。

### CI 质量门禁有什么价值？

它把本地测试变成团队协作里的自动检查。只要提交代码，CI 就会跑 lint、接口测试、覆盖率、meta 测试、前端构建和 Playwright 冒烟。如果核心链路失败或覆盖率低于阈值，流水线就失败。同时 CI artifacts 会保留 Allure、`coverage.xml`、前端构建和浏览器报告，方便失败后定位。

### 覆盖率是不是越高越好？

覆盖率是门槛，不是目标。这个项目设 80% 是为了保证核心代码被测试保护，但真正有价值的是业务断言，比如库存扣减、越权访问、非法状态流转这些场景。高覆盖率不能替代有效断言。

### 这个项目前端有什么用？

前端不是项目重点，但它让演示更直观。面试官可以不用手动调 Swagger，也能看到注册登录、商品列表、创建订单和状态变化。它也证明我能把接口接到真实使用场景里。

### 为什么 UI 自动化只做一条 Playwright 冒烟？

我没有把 UI 自动化做成很重的回归套件，而是只用 Playwright 做一条高价值冒烟链路：页面加载、API 在线、商品展示、注册登录、创建并支付订单、状态筛选和动态质量指标。API 自动化负责业务规则深度，UI 冒烟负责演示链路稳定，并且已经接入 CI。

### DeepSeek 在项目里怎么用？

我把 DeepSeek 做成本地测试设计助手，而不是质量门禁的一部分。`scripts/generate-test-ideas.py` 会读取本地 OpenAPI 契约，整理成 prompt；没有 `DEEPSEEK_API_KEY` 时只输出 prompt，有 key 时才调用 DeepSeek。它用于辅助发现测试点，最终是否加入项目仍然要人工判断，并写成确定性的 Pytest 用例。

### 如果让你继续优化，你会做什么？

我会优先补三类能力：第一是把 Allure 报告发布成 GitHub Pages；第二是进一步补充库存预占、订单超时释放等接近真实电商的库存生命周期；第三是按风险继续扩展少量 UI 冒烟场景，比如订单取消后的库存恢复展示。

## 简历写法

自动化测试方向：

```text
基于 FastAPI + Pytest 搭建订单系统接口自动化测试框架，覆盖登录、商品查询、订单状态、OpenAPI 契约、库存防超卖和并发取消一致性；使用 fixture 管理隔离数据，结合响应校验、数据库断言、Allure、动态覆盖率快照、Playwright 冒烟和 GitHub Actions 门禁，实现接口与浏览器主链路回归自动阻断，并通过 CI 产物保留失败证据；接入 DeepSeek 本地脚本辅助发现测试点。
```

后端开发方向：

```text
基于 FastAPI 开发小型订单管理系统，实现 JWT 鉴权、商品查询、订单创建、支付模拟、取消订单和健康检查接口；使用 SQLAlchemy + SQLite 完成数据持久化，在服务层实现库存扣减、库存恢复、订单状态流转和越权访问控制，并配套自动化回归测试保障接口稳定性。
```

## 讲项目时别踩的坑

- 不要只说“我用了 Pytest”，要说清楚测了哪些业务风险。
- 不要只说“覆盖率 90%+”，要强调覆盖率只是门槛，关键是业务断言。
- 不要把前端说成主要成果，它是演示辅助，核心还是接口自动化和质量门禁。
- 不要说“数据库断言一定比接口断言高级”，正确说法是二者互补。
