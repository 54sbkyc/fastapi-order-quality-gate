# Bug Analysis

这份文档用于说明自动化测试能发现什么类型的问题，以及发现后如何定位和防止回归。面试时可以用它证明：测试不是为了跑绿灯，而是为了保护业务风险。

## 缺陷 1：取消订单没有恢复库存

风险：

- 用户取消订单后，商品库存没有恢复。
- 库存会越扣越少，影响后续用户下单。
- 这类问题只看接口响应可能发现不了。

自动化如何发现：

- 先创建订单，记录创建后的库存。
- 调用取消订单接口。
- 查询数据库中的商品库存。
- 断言库存等于创建订单后的库存加回订单数量。

定位思路：

- 查看 `order_service.cancel_order`。
- 检查是否执行了 `product.stock += order.quantity`。
- 检查事务提交是否遗漏。

回归保护：

- `test_cancel_created_order_success_restores_stock` 持续覆盖该场景。
- 以后修改取消订单逻辑时，CI 会自动阻断库存恢复回归。

## 缺陷 2：用户可以访问其他人的订单

风险：

- A 用户可以查询 B 用户订单。
- 这是典型越权漏洞，影响数据安全。

自动化如何发现：

- 创建 A 用户并下单。
- 创建 B 用户并登录。
- 使用 B 用户 token 查询 A 用户订单详情。
- 断言返回 `FORBIDDEN_ORDER_ACCESS`。

定位思路：

- 查看订单查询服务是否校验 `order.user_id`。
- 查看接口是否从 token 中获取当前用户。
- 查看错误码是否统一。

回归保护：

- `test_user_cannot_access_another_users_order` 覆盖越权访问。
- CI 每次提交都会跑该用例，避免权限判断被误删。

## 缺陷 3：已支付订单还能取消或重复支付

风险：

- 已支付订单被取消，会造成订单状态和库存不一致。
- 重复支付可能导致重复扣款或重复状态更新。

自动化如何发现：

- 创建订单并支付。
- 再次调用取消或支付接口。
- 断言返回 `INVALID_ORDER_STATE`。
- 使用参数化覆盖多组非法状态转换。

定位思路：

- 查看状态机判断是否只允许 `created` 状态执行支付或取消。
- 查看非法状态是否统一抛出业务异常。

回归保护：

- `test_invalid_order_state_transitions_return_business_error` 覆盖非法转换矩阵。
- 参数化让新增非法组合更容易维护。

## 缺陷 4：伪造或过期 token 被接受

风险：

- 攻击者伪造 token 后访问订单接口。
- 过期 token 仍能访问受保护资源。
- token 对应用户已不存在但仍可访问系统。

自动化如何发现：

- 构造格式错误 token。
- 用错误密钥签发伪造 token。
- 构造已过期 token。
- 构造用户不存在但签名合法的 token。
- 分别访问受保护接口，断言返回 `INVALID_TOKEN`。

定位思路：

- 查看 `get_current_user` 是否校验签名和过期时间。
- 查看是否根据 `sub` 查询真实用户。
- 查看异常是否被统一转换为业务错误码。

回归保护：

- 认证负向测试覆盖 token 缺失、伪造、过期和用户不存在。
- 这些测试能帮助说明项目具备基础安全测试意识。

## 缺陷 5：并发下单导致库存超卖

风险：

- 商品只剩 1 件时，两个会话都先读到 `stock=1`。
- 如果代码只在 Python 内存里判断库存并执行 `product.stock -= quantity`，两个会话都可能创建订单成功。
- 最终库存看起来可能是 0，但数据库里实际产生了 2 个订单，这就是超卖。

自动化如何发现：

- 准备一个库存为 1 的商品。
- 打开两个独立 SQLAlchemy Session。
- 两个 Session 都先读取同一个商品，制造陈旧库存读。
- 第一个 Session 创建订单成功。
- 第二个 Session 再创建订单，必须返回 `INSUFFICIENT_STOCK`。
- 最后断言只生成 1 个订单，库存为 0。

定位思路：

- 查看 `order_service.create_order` 是否先读库存再在内存中扣减。
- 检查扣库存是否由数据库条件更新保证，例如 `stock >= quantity`。
- 检查订单创建和库存扣减是否在同一个事务里提交。

回归保护：

- `test_stale_inventory_session_cannot_create_second_order` 覆盖陈旧库存读导致的超卖风险。
- 服务层通过条件更新原子扣减库存，第二个会话会得到 `INSUFFICIENT_STOCK`。

## 缺陷 6：并发取消导致库存重复恢复

风险：

- 两个请求都读取到 `created` 状态后同时取消同一订单。
- 如果只修改内存对象，两个请求都可能把库存加回，导致库存凭空增加。

自动化如何发现：

- 创建订单后，用两个独立 Session 都预先读取同一订单，制造陈旧状态。
- 第一个 Session 取消成功，第二个 Session 再取消必须返回 `INVALID_ORDER_STATE`。
- 最后断言订单只取消一次，库存只恢复一次。

定位与修复：

- 支付和取消都使用 `id + user_id + status=created` 作为数据库条件更新。
- 只有更新行数为 1 的请求可以继续提交，失败请求回滚事务。
- 库存恢复和订单状态在同一事务提交。

回归保护：

- `test_stale_order_session_cannot_restore_stock_twice` 持续覆盖重复恢复风险。

## 缺陷 7：Allure 历史结果污染当前质量数据

风险：

- 本地多次运行后，`allure-results` 不断累积，曾把 46 条实际用例展示成 672 个结果文件。
- 前端手工填写测试数和覆盖率，新增用例后容易与真实报告不一致。

修复与回归保护：

- 质量门禁运行前清理旧 Allure 结果，只保留本次执行证据。
- `scripts/build_quality_snapshot.py` 从当前 `coverage.xml` 和 Allure JSON 生成质量快照。
- React 看板读取快照，Playwright 验证页面指标与 JSON 一致。
- meta 测试保护生成脚本、门禁接线和质量快照结构。

## 缺陷 8：下单超时重试导致重复订单和重复扣库存

风险：

- 服务端已经创建订单，但响应在网络中丢失，客户端无法判断是否成功。
- 客户端直接重试 `POST /api/orders` 会创建第二个订单，并再次扣减库存。
- 两个相同请求并发到达时，仅靠“先查询是否存在”仍可能同时通过检查。

自动化测试设计：

- 相同用户、相同 `Idempotency-Key`、相同参数连续请求两次。
- 断言第一次返回 `201`，第二次返回 `200` 和 `Idempotency-Replayed: true`。
- 查询数据库，断言只有一个订单、一条幂等记录，库存只减少一次。
- 相同键改数量时断言 `409 IDEMPOTENCY_KEY_CONFLICT`。
- 模拟两个请求都未查到记录、后一个写入触发唯一约束冲突，验证事务回滚后返回第一个订单。

修复与回归保护：

- 使用 `(user_id, idempotency_key)` 数据库唯一约束，而不是只依赖进程内判断。
- 在同一事务内完成库存扣减、订单创建和幂等记录写入。
- 捕获唯一约束竞争后先回滚，再读取已提交记录；重复请求不会留下第二次库存扣减。
- 前端网络失败时复用同一个幂等键重试一次，真实 HTTP 回归验证部署后的行为。

## 面试讲法

可以这样总结：

> 我不是只写正常流程测试，而是从业务风险出发设计用例。比如库存恢复、库存超卖、超时重试重复下单、越权访问、非法状态流转和 token 安全，这些都是接口自动化比较适合长期保护的场景。每个问题都有对应的自动化用例，一旦代码变更破坏规则，质量门禁就会失败。
